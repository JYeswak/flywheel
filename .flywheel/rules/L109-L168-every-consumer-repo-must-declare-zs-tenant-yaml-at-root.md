## Rule

Every consumer repository in the ZestStream ecosystem (`mobile-eats`, `alps-insurance`, `vrtx`, `clutterfreespaces`, `terratitle`, `blackfoot-telecom`, `zesttube`, `zeststream-v2-fresh`, `zeststream-platform`, `agent-ui`, `flywheel_gateway`, and **all future projects**) MUST contain a `.zs-tenant.yaml` declaration file at the repository root.

The file declares the repo's identity claim against the canonical `project-mappings.yaml` registry at `~/.claude/skills/infisical-secrets/data/project-mappings.yaml`.

CI MUST verify the declaration matches the registry BEFORE any deploy operation. Build systems (Vercel, Railway, Cloudflare Pages, etc.) MUST invoke the tenant-doctor gate as part of their build command. DB migrations MUST be gated by `zs-verify-supabase-tenant` against the declared Supabase project ref.

Repos lacking `.zs-tenant.yaml` MUST fail deploy with a clear remediation message:
> "Run `/zs:project-bootstrap <slug>` to declare tenant routing for this repo."

## Components

### 1. Per-repo `.zs-tenant.yaml`

Generated by `/zs:project-bootstrap <slug>` skill (skillos-owned canonical at `~/.claude/skills/zs-project-bootstrap/`). Declares:
- Project slug (matches registry row)
- Infisical project ID
- Supabase project ref (if applicable)
- Vercel project ID (if applicable)
- Deploy targets (Railway / Cloudflare / Supabase Edge / etc.)

### 2. Canonical registry

`~/.claude/skills/infisical-secrets/data/project-mappings.yaml` — single source of truth for which credentials route to which tenant. Append-only; rows added when new project bootstraps.

### 3. CI tenant-doctor gate

Build command invokes `pnpm secrets:doctor` (or equivalent) which:
- Reads `.zs-tenant.yaml`
- Pulls Infisical credentials for declared project ID
- Verifies `DATABASE_URL` Supabase ref matches declared ref
- Refuses build on mismatch

### 4. Pre-migration tenant-verification gate (L164 sister)

`zs-verify-supabase-tenant` runs BEFORE `drizzle-kit migrate` / equivalent. Same comparison; refuses on mismatch.
