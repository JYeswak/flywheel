## Rule

Every consumer repository with user-facing surfaces (`mobile-eats`, `alps-insurance`, `vrtx`, `clutterfreespaces`, `terratitle`, `blackfoot-telecom`, `zeststream-v2-fresh`, `zesttube`, and all future projects with client-rendered UI) MUST contain a `.zs-journeys.yaml` declaration file at the repo root.

The file declares **≥3 automated user journeys** per `skillos.user_journey_registry.v1` schema. Each journey is a programmatic walk-through of a real user persona's task (e.g. "hungry local finds lunch within 15 min"; "homeowner files first ALPS claim"; "remote worker activates Blackfoot fiber").

Journeys run automatically in CI via `@zeststream/user-journey-testing` (Playwright-based) on:
- Every PR (against preview deploy)
- Every push-to-main (against staging)
- Nightly cron (against production)

**Production deploys MUST fail if:**
- `.zs-journeys.yaml` is missing
- Fewer than 3 journeys declared
- Any P0-tagged journey fails on the preview deploy

Repos lacking `.zs-journeys.yaml` fail deploy with: `"Run /zs:journey-bootstrap <slug> to declare user journeys for this repo."`

## Why

**Cost-of-violation: SEVERE.** Without journey tests, regressions ship silently. UX-breaking changes reach production undetected: concierge logic drift, broken auth flows, missing CTAs, dead links — all currently invisible to CI. ALPS Insurance, mobile-eats, TerraTitle (legal workflows), Blackfoot Telecom (ISP customer portal) all carry user-facing surfaces with real customer impact.

**Origin (mobile-eats e2e 2026-05-12T~14:00-16:40Z):** Joshua walked through mobile-eats end-to-end and surfaced cumulative product-quality gaps that vendor packaging + safety substrate + placeholder linter + DB-backed data did NOT catch. The system shipped with all infrastructure in place but had **never validated the user could actually accomplish the task the product exists to enable**.

**Joshua's framing:** *"this isn't something I should have to do every time — this needs to be part of our ci/cd process."* Programmatic journey validation removes the operator from the loop while preserving the empathy lens.

**Why fast-track promotion (skipping standard 3-strike):** Joshua-directive PROMOTE-IMMEDIATE. The 4 client repos with regulatory-adjacent surfaces (ALPS / mobile-eats / TerraTitle / Blackfoot) cannot wait for N=3 SATURATION before adopting. Same fast-track pattern as L168 + L169.

## Components

### 1. Per-repo `.zs-journeys.yaml`

Generated by `/zs:journey-bootstrap <slug>` skill (skillos:1 owns canonical at `~/.claude/skills/zs-journey-bootstrap/`). Declares:
- Repo slug + repo type (consumer-app / api / admin / etc)
- ≥3 user journeys, each with:
  - Persona (who is doing this; one sentence)
  - Intent (what they're trying to accomplish; one sentence)
  - Critical path (the click/keystroke/API sequence; programmatic)
  - Expected outcome (what proves success; assertion)
  - Priority (P0 / P1 / P2 — P0 blocks deploy)

### 2. `@zeststream/user-journey-testing` package

skillos-canonical. Playwright-based runner that:
- Reads `.zs-journeys.yaml`
- Executes each journey against the target environment (preview / staging / production)
- Reports pass/fail per journey + timing + screenshot artifacts on failure
- Emits CI annotations + status check

### 3. CI workflow integration

```yaml
# .github/workflows/journeys.yaml (canonical)
name: User Journeys
on: [pull_request, push, schedule]
jobs:
  journeys:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v3
      - run: pnpm install
      - run: pnpm journeys:run --env=${{ github.event_name == 'pull_request' && 'preview' || 'staging' }}
```

### 4. Hard-deploy gate

Vercel/Railway/Cloudflare-Pages build command wraps `pnpm journeys:run --gate`. Exit non-zero → deploy aborts.

## How to apply — per-repo bootstrap sequence

```bash
# Step 1: from repo root, with operator at console:
/zs:journey-bootstrap <slug>

# Skill generates:
#   .zs-journeys.yaml at repo root (scaffold with 3 placeholder journeys)
#   journeys/ directory with Playwright test files (one per journey)
#   package.json: journeys:run + journeys:open scripts
#   .github/workflows/journeys.yaml
#   vercel.json buildCommand wrap (or railway.toml / cloudflare.toml equivalent)

# Step 2: operator fills in the journey details (Joshua-bandwidth)
#   - Edit .zs-journeys.yaml with real persona + intent + critical-path
#   - Run pnpm journeys:open to author the Playwright steps interactively

# Step 3: acceptance
pnpm journeys:run   # all 3+ pass locally
git status          # .zs-journeys.yaml + journeys/ + workflow staged
git commit -m "chore(journey-bootstrap): declare 3 user journeys per L170"
```
