The bead summary listed five "88G/67G/32G/26G/16G" repos as the
remaining storage pressure, but the framing missed the actionable
question: which of those bytes are *regenerable* and which are *real
data*? A worker who deletes 113G of regenerable checkpoints, build
artifacts, and downloadable model weights is doing safe maintenance.
A worker who deletes 82G of `polymarket-pico-z/data` is destroying
work.

The deep-probe pass — `du -sh <repo>/<subdir>` for the top-5 — is
what made the answer crisp. It found that 4 of 5 repos' bytes live
in subdirectories whose contents are by definition reconstructible:
langgraph checkpoints (re-checkpoint on next run), comfyui outputs
(generated, source prompts retained), cargo target (regenerated by
build), third-party model fetches (re-download). Only
`polymarket-pico-z/data` is a real dataset that needs Joshua's
archive-vs-external-vs-prune-vs-capacity call.

Bead conflation worth flagging: the bead body listed
`~/Developer/picoz` as 88G, but `picoz` is currently 0B (cloud-symlink
not present locally — same pattern as ALPS). The actual 88G repo is
`polymarket-pico-z`. Same as the `~/Developer/alpsinsurance` vs
`Desktop/Projects/clients/alps-insurance` reference memory rule: bead
authors trust local probes over remembered paths.

The Tier A / Tier B split is the canonical pattern for
storage-pressure recommendations. Tier A items share a property —
*the action is reversible by re-running a standard command* (cargo
clean, langgraph re-run, fetch script). Tier B items are *decisions
about real data* that need the operator's preference between
permanence trade-offs (archive vs external vs prune vs capacity).
The matrix exposes that distinction explicitly so Joshua can approve
Tier A as an umbrella without having to triage each row.
