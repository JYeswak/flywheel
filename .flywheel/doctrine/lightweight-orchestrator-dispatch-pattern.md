# Lightweight Orchestrator Dispatch Pattern (LODP)

A pragmatic orchestrator dispatch flow that ships ~25 dispatches/hour with
~95% first-pass success rate. Empirically validated 2026-05-08 across
flywheel waves 5-14 (60+ dispatches, 4 codex worker panes).

## When to use this vs the heavyweight /flywheel:dispatch contract

**Use LODP** for routine bead burndown to known-healthy worker panes:
- Codex workers in established sessions with topology rows present
- Dispatch packets generated by `build-dispatch-packet.sh` (which already
  bakes in the 13-block contract: memory hits, skill auto-routes, mission
  fitness, callback contract, file reservations, etc.)
- Workers that honor L120 (worker self-closes bead via `br close` in DONE
  callback)

**Use full `/flywheel:dispatch`** when:
- New session being onboarded
- Mission anchor uncertain (status=drift or unknown)
- Agent Mail courtesy ping required (Step 7)
- Cross-orch coordination dispatch needing audit chain

## The LODP recipe (5 mechanical steps)

```bash
SESSION=flywheel
PANE=<N>
BEAD_ID=<bead-XXX>
TASK_HASH=$(date +%s%N | shasum | cut -c1-6)
TASK_ID="${BEAD_ID}-${TASK_HASH}"
TASK_FILE="/tmp/dispatch_${TASK_ID}.md"

# 1. BUILD — build-dispatch-packet.sh handles the 13-block contract
bash /Users/josh/Developer/flywheel/.flywheel/scripts/build-dispatch-packet.sh \
  --bead-id "$BEAD_ID" --target-pane "$PANE" --target-session "$SESSION" \
  --task-id "$TASK_ID" --apply --json

# 2. LOG — minimal dispatch_sent row (schema_version=2, l140_compliant)
TASK_SHA256=$(sha256sum "$TASK_FILE" | cut -d' ' -f1)
jq -nc \
  --arg ts "$(date -u +'%Y-%m-%dT%H:%M:%SZ')" \
  --arg task_id "$TASK_ID" --arg task_file "$TASK_FILE" \
  --arg task_sha256 "$TASK_SHA256" --arg session "$SESSION" \
  --argjson pane "$PANE" --arg bead "$BEAD_ID" \
  '{schema_version:2, event:"dispatch_sent",
    dispatch_skill_version:"flywheel-dispatch/v2",
    ts:$ts, task_id:$task_id, session:$session, pane:$pane,
    topology_resolved_pane:$pane, topology_role:"worker",
    callback_session:$session, callback_pane:1,
    task_file:$task_file, bead_id:$bead,
    agent_type:"codex", worker_substrate:"codex-pane",
    pane_state:"WAITING", capture_provenance:"live",
    mission_anchor:"<your-mission-anchor>",
    mission_fitness_class:"adjacent",
    task_sha256:$task_sha256, edits_expected:true,
    dispatch_status:"queued_for_send",
    l140_compliant:true}' \
  >> "${REPO}/.flywheel/dispatch-log.jsonl"

# 3. SEND — dispatch-and-verify.sh (post-erkab multi-signal classifier)
bash /Users/josh/Developer/flywheel/.flywheel/scripts/dispatch-and-verify.sh \
  "$SESSION" "$PANE" "$TASK_FILE"

# 4. RECOVER — if codex chevron stuck (state=WAITING after send), Enter:
printf 'y\n' | ntm send "$SESSION" --pane=$PANE ""

# 5. REAP — when DONE callback arrives, worker has self-closed via L120
#          (br_close_executed=yes); orchestrator commits artifacts:
git add <worker-touched-paths>
git commit -m "feat(<scope>): land <bead-id> — <one-line>"
```

## The SCOPE_EXPANSION_REQUEST protocol

Workers honoring scope discipline will ask for explicit file approval when
the dispatch packet doesn't name exact paths. Pattern:

```
Worker → Orch:  SCOPE_EXPANSION_REQUEST <task-id>
                 requested_paths=<comma-separated>
                 reason=<one-line>

Orch → Worker:  SCOPE_EXPANSION_APPROVED <task-id>
                 approved_paths=<comma-separated>
                 reason=<why approval is data-decided>
```

Approval is data-decided per L70 (NO-PUNT) — if files match bead intent
and stay within repo scope, approve and let work proceed. Don't gate on
operator unless TRUE-blocker class.

## Idle pane = L70 NO-PUNT failure

When a worker callback closes a bead, the pane returns to WAITING. The
orchestrator MUST dispatch the next bead immediately, not wait for next
tick boundary or operator prompt. Idle panes are non-accretive ticks.

The orchestrator's job between callbacks:
1. Reap the closed callback (commit artifacts if not already)
2. Pick next bead from `br ready --json --limit 10`
3. Build packet + send via dispatch-and-verify.sh
4. Repeat

Total time idle-pane → next-dispatch should be <30 seconds.

## Anti-patterns

| Anti-pattern | Why it fails | Fix |
|---|---|---|
| Wait for callback before dispatching next pane | Serializes work; idle workers waste capacity | L70 — dispatch on idle, not on call-and-response |
| Trust `dispatch-and-verify` exit code as truth | Codex slow-start (>30s warmup) produces false-negative STUCK | Inspect pane scrollback if rc=1 but status looks recoverable |
| Run full canonical-cli precheck (Step 3b) on every dispatch | Heavy; build-dispatch-packet.sh already validates | Only run for new CLI surface dispatches |
| Send agentmail courtesy ping (Step 7) on every dispatch | Spams audit log; not required for routine work | Skip for in-session burndown; use for cross-orch only |
| Run two-truth-sources validator (Step 1b) on every dispatch | False positive on codex prepared chevron (L142) | Skip if pane is known-healthy + capture_provenance=live |
