## 1. Why this exists

Joshua, 2026-05-20:
> *"I need to be able to approve / deny repo wide insight from a single location."*

Before this surface, orchestrators interrupted Joshua's user pane (xpane to pane 0) every time a
Tier-3 decision arose. With 11 orchs, that produced 15+ xpane interruptions/day. Joshua read
them out of order, lost context, and the orchs blocked on his inline reply.

With Josh-Review, every Tier-3 decision is a file in `~/Josh-Review/pending/`. Joshua moves the
file to `approved/`, `denied/`, or `archived/` at his cadence. Orchs poll on tick and act.

## 2. Folder shape (mirrors `~/Josh-Review/_README.md`)

```
~/Josh-Review/
├── _README.md          # canonical mobile-eats authored README — source of truth for shape
├── _INDEX.md           # auto-generated by josh-review-index-builder.sh (30min cron)
├── pending/            # awaiting Joshua review
├── approved/           # Joshua approved → repo orch acts → append acted-on ledger → archive
├── denied/             # Joshua denied → repo orch backs off → archive
└── archived/           # post-action archive (orch responsibility)
```

## 3. Filing bar (the only things that get filed)

A file goes in `pending/` ONLY if it is:

- **Tier-3 decisions** — scope, taste, paid actions, security boundaries
- **Cross-repo paradigm conflicts** — affects ≥2 repos (e.g., mobile-eats expo-vs-capacitor)
- **Doctrine adoption** affecting multiple repos in the fleet

What does NOT get filed (autonomous instead):

- Routine dispatch ("pick next bead?") — orch decides
- Status updates ("worker done") — dispatch log
- Doctrine sync ("pull this") — skill files propagate via next-CC-session, not Joshua's inbox
- Emergencies — xpane still allowed (time-critical only)

**Filing requirement:** every pending file MUST contain a `Why-not-autonomous:` line declaring
why the orch could not decide itself. Files lacking this line are rejected by the poll skill.

## 4. Rate caps (guardrails against inbox flood)

```
Per-orch cap:     3 pending at once  (4th attempt → orch must self-resolve or wait)
Fleet-wide cap:   25 pending total
Aging policy:     72h → yellow (urgent)  |  7d → auto-deny candidate
Format cap:       1 page (≤ ~80 lines after the YAML header)
```

The auto-index builder surfaces:
- Items aged > 72h with a yellow indicator
- Items aged > 7d as auto-deny candidates (Joshua still moves; we do not move files)

## 6. Per-orch poll behavior

Every orchestrator's `/flywheel:tick` includes a non-blocking poll step:

```bash
.flywheel/scripts/josh-review-poll.sh --json
```

This emits:
- A dashboard line to stderr: `📁 Josh-Review: X approved-pending-action, Y denied-pending-ack`
- A JSON receipt to stdout with `josh_review_pending_action` for doctor consumption
- `approved_pending_action[]` — files Joshua moved to `approved/` that this orch has not yet acted on
- `denied_pending_act[]` — files moved to `denied/` not yet acknowledged

"Acted on" is tracked in `~/.local/state/flywheel/josh-review/acted-on.jsonl` (append-only;
keyed by file sha256). The poll step is **read-only** against `~/Josh-Review/`. The orch (or
worker dispatched by the orch) appends to the ledger after acting; cleanup-to-archive is a
separate sweep, not blocked by the poll.

## 7. Auto-index cron

`~/Library/LaunchAgents/ai.zeststream.josh-review-index.plist` runs
`.flywheel/scripts/josh-review-index-builder.sh` every 30 minutes. It rebuilds
`~/Josh-Review/_INDEX.md` with:

- Top-10 pending items grouped by repo
- Items sorted by `priority_weight × age_hours`
- Yellow indicator on items > 72h
- Auto-deny candidate flag on items > 7d
- TL;DR excerpt per item

## 8. flywheel-init integration

New repos adopting flywheel get a one-liner appended to their tick recipe:

```bash
[ -x .flywheel/scripts/josh-review-poll.sh ] && .flywheel/scripts/josh-review-poll.sh --quiet
```

The flywheel-init bootstrap hook (`.flywheel/scripts/josh-review-init-hook.sh`) is invoked by
`/flywheel:adopt` and `/flywheel:init` to drop the poll script + tick wire-in into the new repo.

## 9. xpane spam ban (paired doctrine)

This surface only works if orchs STOP doing what we did before (xpane "doctrine landed; pull
when ready"). The paired ratified doctrine is in skill
`~/.claude/skills/agent-friendly-doctrine-propagation/` and bans:

- Fleet propagation announcements via xpane
- "Pull this when ready" broadcasts
- Adoption requests

xpane is still allowed for: active worker callbacks, substrate-down emergencies, cross-orch
handoffs blocking on a peer reply, replies to peer ASKs.
